# 川渝医护招聘爬虫新手维护指南

这份文档给第一次接手项目的同事用，目标只有两个：

1. 先把程序跑通，知道正常产物长什么样。
2. 学会新增医院、测试选择器、筛出有用的最新招聘，并在需要时继续做详情增强。

如果你还要处理微信公众号 RSS 订阅、公众号招聘文章筛选、共享草稿串推送，请再配合阅读隔壁项目 [Wechat_api/wechat-download-api/新手操作与维护指南.md](/Users/xiongtao/Documents/scaper_zhaopin/Wechat_api/wechat-download-api/新手操作与维护指南.md)。

## 1. 这个项目是做什么的

`chongqing_medical_jobs_crawler` 负责抓医院官网招聘列表页，把新公告增量保存到本地数据库，并导出“本次新增招聘清单”。

它的典型工作链路是：

列表页提取 -> 招聘关键词过滤 -> 增量去重入库 -> 导出本次新增 -> 可选详情增强 -> 生成公众号待审包

和 `wechat-download-api` 的分工是：

- 本项目更偏“医院官网招聘采集”和“详情页处理”。
- `wechat-download-api` 更偏“微信公众号 RSS 拉取、招聘筛选、导出 confirmed 招聘、推公众号草稿”。

## 3. 第一次跑通

建议第一次严格按下面顺序跑。

### 第一步：跑基础测试

```bash
python test_basic.py
```

这个脚本主要检查：

- 数据库初始化是否正常
- 配置是否能加载
- 招聘过滤规则是否生效
- 导出逻辑是否可用

如果这里就报错，先不要急着跑全量爬虫，优先看错误信息和 `logs/crawler.log`。

### 第二步：只联调 1 个站点

```bash
python main.py --all --site-limit 1
```

这样做的好处是：

- 能最快确认主流程能跑
- 不会一下子跑很多站点，排错成本低
- 如果有报错，更容易定位是哪一个站点的问题

### 第三步：查看数据库统计

```bash
python main.py --stats
```

如果前两步正常，你通常能看到：

- 数据库里已有职位数
- 最近 30 天职位数
- 各站点已保存职位数

### 第四步：看本次导出产物

主流程跑成功后，通常会在 `data/exports/` 下看到两类文件：

- `new_jobs_时间戳.csv`
  用来查看本次新增职位明细
- `wechat_digest_时间戳.md`
  用来快速浏览“本次新增招聘清单”

如果本次没有新增职位，程序会提示未生成导出文件，这不是错误。

## 4. 新增医院并测试跑通

新增医院的核心入口是：

- 配置文件：`医院招聘的selector.csv`
- 诊断脚本：`test_hospital_url_interactive.py`
- 单站测试：`python main.py --test ...`
- 单站试跑：`python main.py --site ...`

### 4.1 CSV 六列分别是什么意思

`医院招聘的selector.csv` 当前表头是：

```text
hospital_name,list_page_url,base_url,listing_selector,list_title_selector,link_selector
```

各列含义：

- `hospital_name`
  医院名称。建议写全称，后面可以直接用医院名运行 `--site`。
- `list_page_url`
  招聘公告列表页 URL。这一列最重要，测试时就是围绕它排查。
- `base_url`
  站点基础域名。用于把相对链接补成绝对链接。
- `listing_selector`
  列表项选择器，要能命中每一条公告的外层节点。
- `list_title_selector`
  标题选择器，要能从列表项中拿到公告标题文本。
- `link_selector`
  链接选择器，要能拿到详情页链接的 `href`。

### 4.2 填写规则

新手维护时建议遵守这几个规则：

- `list_page_url` 必须是能直接打开的 `http://` 或 `https://` 地址。
- `base_url` 最好只保留协议和域名，例如 `https://www.example.com/`。
- `listing_selector` 先保证“能提取到公告项”，再考虑写得多优雅。
- `list_title_selector` 和 `link_selector` 优先从同一条列表项内部取值。
- 如果标题和链接都在同一个 `<a>` 上，这两列通常都可以写 `a` 或更具体的选择器。
- 如果站点是前端壳页、接口渲染页或带强 WAF，不要死磕 CSS 选择器，先用下面的诊断脚本判断分类。

### 4.3 推荐调试顺序

#### 第 1 步：先补一行 CSV

把新医院加到 `医院招聘的selector.csv`。

建议一开始只补最小可用信息，不要同时改很多医院。

#### 第 2 步：用交互脚本粘贴列表页 URL 诊断

```bash
python test_hospital_url_interactive.py
```

然后把医院列表页 URL 粘进去。

这个脚本会输出：

- 诊断分类
- 建议动作
- 匹配到的医院
- CSV 里的列表页
- 抓取后端
- 提取到的前几条招聘公告

它很适合第一时间判断问题是：

- 选择器没命中
- 页面本身被 WAF/TLS 拦住
- 页面是前端壳页
- 页面数据来自接口

#### 第 3 步：用主程序的 `--test` 看样本数据

可以按医院名，也可以按 URL：

```bash
python main.py --test "重庆市人民医院" -v
python main.py --test "https://example.com/recruit/list" -v
```

`--test` 不会入库，适合确认：

- 当前配置是否能提取到招聘列表
- 日志里展示的样本标题和链接是否正确
- 诊断分类是否正常

#### 第 4 步：做单站试跑

```bash
python main.py --site "重庆市人民医院"
```

如果你不确定医院名是否写对，也可以直接用列表页 URL：

```bash
python main.py --site "https://example.com/recruit/list"
```

单站试跑的目标是确认：

- 能正常提取
- 能通过招聘过滤
- 能入库
- 能生成本次新增导出

#### 第 5 步：需要详情时再做详情增强

```bash
python main.py --site "重庆市人民医院" --with-details --select-detail-urls
```

这一步适合在列表页已经稳定后再做，不建议把“列表页调试”和“详情增强调试”混在一起。

## 5. 这个项目怎么得到“有用的最新招聘”

很多新手会问两个问题：

1. 为什么明明抓到很多公告，最后入库的不多？
2. 为什么有些公告被过滤掉了？

核心原因是这个项目不是“看到公告就全收”，而是会先做招聘过滤。

### 5.1 实际链路

程序当前的主链路是：

1. 从医院列表页提取标题和链接
2. 使用 `config/global.yaml` 里的 `recruitment_filter` 做关键词保留/排除
3. 通过 URL 做增量去重，已存在的职位默认不重复入库
4. 把本次新增职位写入 `data/jobs.db`
5. 导出本次新增 CSV 和 Markdown 清单
6. 如果加了 `--with-details`，再继续抓正文、图片、附件，生成详情文章包和公众号待审包

### 5.2 为什么有些公告会被过滤掉

当前过滤规则定义在：

- `config/global.yaml`
- 运行代码里对应 `core/crawler.py` 的 `filter_jobs`

保留词示例：

- `招聘`
- `招考`
- `人才`
- `护士`
- `规培`
- `录用`

排除词示例：

- `招标`
- `采购`
- `比选`
- `设备`
- `耗材`
- `试剂`

也就是说，下面这些情况很常见：

- 真招聘公告会被保留
- 采购、比选、招标类公告会被过滤掉
- 某些标题太模糊、没有明显招聘关键词，可能会被漏掉

如果你发现“该保留的没保留”，优先检查：

1. 标题是否真的包含招聘语义
2. `config/global.yaml` 里的保留词是否需要补充
3. 选择器是否取错了标题文本

## 6. 详情增强和待审包

当你不只是想拿到列表，还想进一步处理正文、图片、附件时，再用详情增强。

### 6.0 从爬虫到公众号草稿箱的完整操作流程

这部分是很多新手最容易混淆的地方。正确顺序不是“先手动找审核 URL，再回去跑主程序”，而是下面这条链路：

#### 流程总览

1. 先运行主程序，抓到本次新增招聘
2. 如果你带了 `--select-detail-urls`，程序会在详情增强前自动启动本地审核页
3. 你在浏览器里审核“哪些详情 URL 继续处理”
4. 点击页面底部“完成并继续”后，命令行会自动恢复
5. 程序继续抓取保留下来的详情页，生成详情文章包和 `wechat_review` 待审包
6. 如果待审包里有附件，需要时先运行附件上传脚本回填可用链接
7. 最后把单个 `review_dir` 或整批 `summary.json` 推送到公众号官方草稿箱

也就是说：

- “审核页面”不是你先手动运行的独立程序
- 它是在 `main.py --with-details --select-detail-urls` 这条命令里自动启动的
- 你审核完成后，主程序会自动继续跑，不需要再手动回去执行第二次详情抓取

#### 最推荐的新手操作方式

```bash
python main.py --site "重庆市人民医院" --with-details --select-detail-urls
```

这条命令会依次做三件事：

1. 先抓这个医院本次新增的招聘列表
2. 自动打开本地审核页，让你确认哪些详情 URL 值得继续抓
3. 审核完成后自动生成详情包和公众号待审包

#### 审核页是怎么出现的

当你带上 `--select-detail-urls` 时：

- 程序会先在 `data/exports/detail_review/时间戳/` 下生成审核材料
- 然后启动一个本地审核页面
- 默认配置下会自动尝试打开浏览器
- 日志里会显示审核页面地址

你要做的是：

1. 点开原文确认是不是你想要的“最新招聘”
2. 对每条候选做保留、丢弃或稍后决定
3. 点击页面底部的“完成并继续”

点击后，命令行会自动恢复，不需要重新执行主程序。

#### 审核结束后会得到什么

审核结束并继续处理后，常见产物有两类：

- `data/exports/details/时间戳/summary.json`
  这是本轮详情增强的总汇总，后面可以直接拿去做批量推送
- `data/exports/wechat_review/时间戳/文章目录/`
  这是单篇公众号待审包目录，也常叫 `review_dir`

你后面推送公众号草稿箱，核心就是使用这两个产物里的一个：

- 推单篇：用 `review_dir`
- 推整批：用 `summary.json`

#### 什么时候需要先上传附件

不是所有待审包都要先跑附件上传。

判断方法：

- 如果待审包里没有附件，可以直接推草稿箱
- 如果待审包里有附件，并且你希望文章里回填可用的附件链接或小程序路径，先跑附件上传脚本

单篇附件上传示例：

```bash
python upload_wechat_attachments.py --review-dir "/absolute/path/to/data/exports/wechat_review/<时间戳>/<文章目录>"
```

这个脚本会做几件事：

- 读取待审包里的附件
- 上传到指定服务
- 回填 `package.json`
- 更新 `article.html`
- 生成 `attachment_links.json` 和 `attachment_links.md`

它跑完后，日志里也会提示下一步可以继续执行：

```bash
python push_wechat_review_to_official_draft.py --review-dir "/absolute/path/to/review_dir"
```

#### 最后怎么推送到公众号草稿箱

有两种常见方式。

方式 A：推送单篇待审包

```bash
python push_wechat_review_to_official_draft.py --review-dir "/absolute/path/to/data/exports/wechat_review/<时间戳>/<文章目录>"
```

方式 B：按本轮详情增强汇总批量推送

```bash
python push_wechat_review_to_official_draft.py --summary-path "/absolute/path/to/data/exports/details/<时间戳>/summary.json"
```

理解这两个参数很重要：

- `--review-dir`
  适合你手动确认只推某一篇
- `--summary-path`
  适合一轮详情增强结束后，按顺序把这一批可用待审包一起推进去

如果你只记一条完整流程，可以记下面这个版本：

```bash
# 1. 主程序抓列表 + 自动进入审核 + 自动继续详情增强
python main.py --site "重庆市人民医院" --with-details --select-detail-urls

# 2. 如有附件，先回填附件链接
python upload_wechat_attachments.py --review-dir "/absolute/path/to/data/exports/wechat_review/<时间戳>/<文章目录>"

# 3. 推送单篇待审包到公众号草稿箱
python push_wechat_review_to_official_draft.py --review-dir "/absolute/path/to/data/exports/wechat_review/<时间戳>/<文章目录>"
```

如果你做的是整批处理，把第 3 步改成 `--summary-path` 即可。

#### 如果你不想再手动找路径

现在可以直接用这个一键脚本，默认自动处理“最新一轮审核结果”：

```bash
./venv/bin/python publish_latest_wechat_batch.py
```

它会自动做这些事：

1. 找到最新的 `data/exports/details/<时间戳>/summary.json`
2. 读取这一批里的所有 `review_dir`
3. 某篇有附件时，先自动上传附件并回填 `article.html` / `package.json`
4. 然后把这一批待审包依次推送到共享公众号草稿串

也就是说，你审核完之后，通常直接跑这一条就行，不用自己再查路径。

## 8. 常见问题排查

### `selector_mismatch`

含义：

- 页面拿到了，但当前选择器没有命中正确的公告项

优先检查：

- `listing_selector`
- `list_title_selector`
- `link_selector`

建议动作：

- 先在浏览器开发者工具里确认真正的列表节点
- 不要直接复制最外层大容器
- 用 `test_hospital_url_interactive.py` 反复验证

### `empty_selector`

含义：

- CSV 里缺少可用的 `listing_selector`

建议动作：

- 先补可用的列表项选择器
- 如果页面明显是纯动态页，再考虑接口或浏览器兜底，不要留空硬跑

### `waf_or_tls_blocked`

含义：

- 页面更像被 WAF、证书、TLS 或上游网关问题拦住了

常见现象：

- 403、429、503、504
- 返回验证码页、拦截页、网关错误页

建议动作：

- 先确认浏览器能不能正常打开
- 看是不是证书异常或域名跳转异常
- 再看是否需要放宽某些站点的 SSL 校验或使用动态抓取链路

### `frontend_shell`

含义：

- 抓到的更像前端壳页，真正数据可能是浏览器运行后才加载出来

建议动作：

- 打开浏览器开发者工具，看页面渲染后的 DOM
- 检查 Network 里是否有列表接口
- 优先找接口，不要先上复杂选择器

### `api_backed`

含义：

- 页面数据大概率来自接口

建议动作：

- 直接抓浏览器 Network 请求
- 找返回公告 JSON 的接口
- 评估是否补站点级 API 兜底逻辑，而不是继续调 HTML 选择器

2. 再跑 `python main.py --all --site-limit 1`
3. 确认 `data/exports/` 有导出结果
4. 新增医院时只改一行 CSV
5. 用交互诊断脚本确认列表页是否可提取
6. 用 `--test` 验证样本数据
7. 用 `--site` 做单站试跑
8. 列表页稳定后再考虑 `--with-details`

照这个顺序做，排错成本会低很多。
