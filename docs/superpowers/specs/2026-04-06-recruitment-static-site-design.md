# 招聘公告静态站设计

**目标**

把本地数据库里的招聘公告导出成一个可部署到 GitHub / Cloudflare Pages 的纯静态网页，面向普通用户展示：

- 标题
- 发布日期
- 原文入口

其中原文链接不直接展示 URL，而是显示为 `阅读原文` 按钮。用户点击后先看到公众号引导弹层，弹层中展示现有公众号图片 `/static/official_wx_card_qr.jpg`，再由用户手动点击“我已关注，继续查看”跳转到原公告页面。

**范围**

- 首版只覆盖本地数据库导出，不接服务器自动推送。
- 首版筛选数据范围固定为：
  - `is_recruitment = 1`
  - `review_status = confirmed`
  - 最近 30 天
- 首版提供标题关键词搜索，不做来源筛选、分页、排序切换和详情页。
- 首版必须优先适配手机，保证用户容易搜索、容易点按钮、容易完成跳转。

**不做的事**

- 不尝试判断用户是否真的已经关注公众号。
- 不引入后端接口参与页面运行。
- 不在页面上直接暴露原始 URL 字符串。
- 不在首版实现服务器自动推送到本地电脑。

**现状**

- 仓库已有本地 SQLite 数据库：`/Users/xiongtao/Documents/zhaop_automotion/data/rss.db`
- `articles` 表已经包含首版所需核心字段：
  - `title`
  - `link`
  - `publish_time`
  - `is_recruitment`
  - `review_status`
- 现有仓库里可直接复用公众号图片：
  - `/Users/xiongtao/Documents/zhaop_automotion/static/official_wx_card_qr.jpg`
- 当前库里满足“最近 30 天 + `confirmed` + 招聘”的记录量约为 69 条，适合先做纯静态测试。

**推荐方案**

采用“本地数据库单向导出 + 静态 HTML 读取静态 JSON”的方式：

1. 本地新增一个生成命令，从 `rss.db` 查询符合条件的招聘公告。
2. 生成一个静态站点输出目录，固定使用 `site/`。
3. 站点目录中至少包含：
   - `index.html`
   - `recruitment.json`
   - 公众号图片副本
   - 少量 CSS / JS 资源
4. 页面运行时只读取本地生成的 `recruitment.json`，不依赖 FastAPI 或其他后端服务。
5. 用户更新数据时，只需要在本地重新运行生成命令，再推送到 GitHub，由 Cloudflare 发布。

这条路径最符合当前部署方式，也为后续“服务器数据先进入本地数据库，再统一重新导出静态站”留出了清晰扩展点。

**为什么不用单 HTML 内嵌数据**

- 单 HTML 虽然文件更少，但后续维护和更新都更笨重。
- 把页面结构和数据拆开后，后面接入更多公告数据时更容易管理。
- 浏览器端关键词过滤读取 JSON 更直观，也方便调试。

**总体结构**

本次实现固定采用以下职责分层：

- 一个导出脚本
  - 负责查询 SQLite
  - 负责筛选、排序、格式化日期
  - 负责生成 `recruitment.json`
  - 负责复制二维码图片到静态输出目录
- 一个静态页面入口
  - 负责加载 JSON
  - 负责渲染公告列表
  - 负责标题关键词搜索
  - 负责弹出公众号引导层
  - 负责最终跳转原文
- 一个固定输出目录
  - 作为 GitHub / Cloudflare Pages 的部署内容来源

**数据流**

首版数据流固定为：

`本地 rss.db -> 本地导出脚本 -> recruitment.json + 静态资源 -> Git 提交/推送 -> GitHub / Cloudflare Pages`

后续扩展数据流固定保持为：

`服务器推送或同步 -> 本地 rss.db -> 本地导出脚本 -> 静态站发布`

关键原则是：静态页面永远只认导出结果，不直接连接数据库，也不直接连接服务器。

**导出规则**

导出脚本需要执行以下规则：

1. 从 `articles` 表读取记录。
2. 只保留：
   - `is_recruitment = 1`
   - `review_status = 'confirmed'`
   - `publish_time` 在最近 30 天内
3. 按 `publish_time` 倒序输出。
4. 导出字段固定为：
   - `title`
   - `url`
   - `publish_date`
   - `publish_timestamp`
5. 输出 JSON 时保持前端友好格式，避免前端再做复杂转换。

导出后的 JSON 结构固定如下：

```json
{
  "generated_at": "2026-04-06 16:00:00",
  "count": 69,
  "items": [
    {
      "title": "某医院 2026 年公开招聘工作人员公告",
      "url": "https://example.com/article/123",
      "publish_date": "2026-04-05",
      "publish_timestamp": 1775328000
    }
  ]
}
```

**页面结构**

页面固定为单页结构，避免引入多页面跳转成本：

1. 顶部头图区
   - 页面标题
   - 一行简短说明
   - 明确告诉用户点击原文前会先看到公众号提示
2. 搜索区
   - 一个大号输入框
   - 一个搜索按钮
   - 支持输入时即时过滤
3. 公告列表区
   - 单条公告卡片只展示标题、发布日期、阅读原文按钮
4. 空状态区
   - 搜索无结果时给出明确提示
5. 弹层区
   - 默认隐藏
   - 点击阅读原文后显示二维码、说明文案和继续按钮

**交互设计**

页面交互固定为“列表优先，二维码重点放在弹层里”：

- 用户进入页面后先看到招聘列表，而不是先被大幅二维码占据首屏。
- 点击任意一条 `阅读原文` 时，页面记录当前目标链接并打开弹层。
- 弹层内展示：
  - 公众号图片
  - “请先关注公众号，再继续查看原文”的提示文案
  - `我已关注，继续查看`
  - `关闭`
- 用户点击继续后，在新标签页打开对应原文。
- 用户关闭弹层后返回列表，当前滚动位置保留。

这种方式的优点是：

- 首页更像招聘信息服务，而不是宣传页。
- 用户已经对某一条公告感兴趣时再触发引导，接受度更高。
- 对首屏空间更友好，尤其适合手机端。

**手机优先设计**

手机体验是首版重点，页面应优先从小屏出发设计，而不是桌面端缩小：

- 列表采用单列卡片布局。
- 搜索框占满可用宽度，输入区高度足够，方便手指点击。
- `阅读原文` 按钮使用整行或接近整行的宽按钮，避免误触。
- 标题字号、行高和按钮间距优先保证可读性和可点性。
- 弹层在手机端使用更靠近底部或居中的大面板，不做过小的浮层。
- 弹层按钮纵向堆叠，主按钮明显高亮。
- 页面避免横向滚动和过密信息堆叠。

首版验收时应从真实用户点击体验判断是否达标，而不是只看浏览器自适应是否“能显示”。

**桌面端表现**

桌面端在不破坏手机优先原则的前提下增强层次：

- 更宽的内容容器
- 列表卡片保留足够留白
- 标题和说明区域更舒展
- 可在首屏轻量提示“阅读原文前需先关注公众号”

但桌面端不应额外增加复杂侧栏或多列信息，以免和手机端体验分叉过大。

**视觉方向**

首版视觉固定为清晰、可信、偏信息服务型，而不是做得过于营销化：

- 页面以招聘列表为主角
- 公众号二维码只在弹层中高亮呈现
- 按钮颜色需要明确区分主次动作
- 发布日期信息弱于标题，但要足够清楚
- 搜索框和卡片边距要让手机端有舒适点击区

**可访问性和易用性**

首版至少保证以下基础体验：

- 键盘可聚焦搜索框和按钮
- 弹层打开后可明显看到关闭入口
- 按钮文字直白，避免让用户猜测下一步
- 外链使用新标签页打开，避免用户完全离开列表页
- 搜索为空时展示全部数据
- 搜索无结果时有明确提示，不出现空白页面

**目录结构**

静态输出目录固定为：

- `site/index.html`
- `site/recruitment.json`
- `site/assets/official_wx_card_qr.jpg`
- `site/assets/site.css`
- `site/assets/site.js`

**命令形态**

首版需要一个你能手动重复执行的生成命令，目标是“一条命令完成站点刷新”。

例如：

```bash
python3 scripts/export_recruitment_static_site.py
```

脚本完成后应直接刷新静态输出目录内容，避免要求你手动拷文件。

**后续扩展方式**

以后接入服务器推送数据时，不改静态页面代码，而是继续扩展数据库入库链路：

1. 服务器将新增 RSS 数据推送或同步到你本地电脑。
2. 本地程序把这批数据写入同一个 `rss.db`。
3. 导出脚本再次运行，重新生成静态站。

这样能保证：

- 页面逻辑保持稳定
- 发布流程固定
- 问题定位更清楚，能区分是“数据没进库”还是“页面没更新”

**边界与失败处理**

- 如果数据库为空：仍生成站点，但页面显示“暂无招聘公告”。
- 如果 30 天内没有符合条件的数据：页面显示空状态，不算脚本失败。
- 如果二维码图片缺失：脚本应明确报错，避免生成半成品站点。
- 如果 JSON 生成成功但页面资源缺失：视为生成失败。
- 如果原文 URL 为空：该条记录不导出，避免产生无法跳转的按钮。

**测试与验收**

首版验收聚焦以下内容：

1. 本地生成命令能成功生成静态站目录。
2. 页面能展示最近 30 天、`confirmed` 的招聘公告。
3. 页面支持按标题关键词搜索。
4. 页面不直接显示原始 URL，只显示 `阅读原文`。
5. 点击 `阅读原文` 时先出现公众号弹层。
6. 弹层中能显示 `/Users/xiongtao/Documents/zhaop_automotion/static/official_wx_card_qr.jpg` 对应图片内容。
7. 点击 `我已关注，继续查看` 后，原文在新标签页打开。
8. 手机端无横向溢出，搜索框、按钮、弹层都方便手指点击。
9. 搜索无结果时有明确空状态提示。
10. 首版不依赖 FastAPI 在线服务即可运行和部署。

**实施结果**

本设计批准后，实施阶段至少应交付：

- 一个本地静态站导出脚本
- 一个静态站输出目录
- 一个手机优先的招聘列表页面
- 一个读取 `recruitment.json` 的前端脚本
- 一套本地验证命令
- 一段简短部署说明，告诉维护者如何生成并推送到 GitHub / Cloudflare
