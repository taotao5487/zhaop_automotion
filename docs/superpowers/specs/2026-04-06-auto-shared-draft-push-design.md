# 自动共享草稿串推送设计

**目标**

在保留现有手动推送能力的前提下，让服务器每天北京时间 `10:00` 和 `16:30` 自动调用现有共享草稿串推送接口，把符合推送条件的招聘文章追加到微信草稿串，并通过企业微信机器人发送结果通知。

**范围**

- 自动任务通过服务器 `cron` 调用现有 API，而不是把定时逻辑塞进主服务进程。
- 共享草稿串的“未满 8 篇则追加，满了再创建新串”继续复用现有后端逻辑，不重复实现。
- 自动任务需要覆盖三种结果：
  - 推送成功：发送详细通知
  - 没有可推送文章：发送简短通知
  - 推送失败：发送错误通知
- 手动 `curl -X POST "/api/recruitment/official-draft/series/push?all=true"` 仍然保留，和自动任务并存。

**现状**

- 主服务已经提供共享草稿串批量推送接口：`POST /api/recruitment/official-draft/series/push?all=true`
- 招聘文章筛选已经支持推送时间窗和订阅后发布时间过滤。
- 企业微信机器人 webhook 已经可用，适合复用为自动任务结果通知通道。
- 主服务当前没有适合直接扩展的“通用自动化任务中心”，因此 cron 更简单、稳定、可维护。

**推荐方案**

采用“`cron` + 本地 shell 脚本 + 调现有 API + 复用企业微信 webhook”的方式：

1. 新增一个服务器可执行脚本，负责：
   - 读取 `.env`
   - 调用共享草稿串批量推送接口
   - 解析返回 JSON
   - 向企业微信机器人发送成功/无文章/失败通知
2. 通过 `crontab` 在每天 `10:00` 和 `16:30` 执行该脚本。
3. 手动推送继续直接调原接口，不做兼容性改动。

**为什么不把定时逻辑写进主服务**

- 主服务已经有 RSS 轮询和预览 worker，再塞一个定时推送任务会让生命周期更复杂。
- `cron` 调 API 更容易独立调试；即使自动任务坏了，主服务仍然可用。
- 对新手维护更友好：问题可以拆成“接口正常吗”和“cron 正常吗”两层排查。

**脚本职责**

脚本需要：

- 从 `.env` 读取：
  - `SITE_URL`
  - `WEBHOOK_URL`
- 调用：
  - `POST ${SITE_URL}/api/recruitment/official-draft/series/push?all=true`
- 处理三种结果：
  - `success=true` 且 `processed_count > 0`
  - `success=true` 但 `processed_count == 0` 或接口返回“没有未推送文章”
  - HTTP 失败 / JSON 失败 / 接口返回 `success=false`

**通知设计**

通知统一发到企业微信机器人，尽量简洁。

成功通知包含：

- 触发时间
- 批量模式结果
- 推送篇数
- 新建草稿串数量
- 草稿 `media_id` 列表
- 结果状态统计

无文章通知包含：

- 触发时间
- “本次无可推送招聘文章”

失败通知包含：

- 触发时间
- 调用失败原因或接口错误文本

**手动与自动并存**

- 手动推送仍然由维护者直接运行 curl。
- 自动任务只是定时调用同一个接口。
- 因为接口本身已经基于“未推送文章”筛选，所以自动和手动不会重复推送同一篇未修改文章。

**边界与失败处理**

- 如果主服务不可达：脚本发送失败通知。
- 如果 webhook 未配置：脚本仍执行推送，但只输出日志到终端/cron 邮件。
- 如果接口返回“当前没有未推送到共享草稿串的 confirmed 招聘文章”：按“无文章”处理，不算失败。
- 如果同一时刻人工也在推送：沿用现有共享草稿串租约/状态控制，不单独发明锁。

**测试与验收**

实现后需要验证：

1. 手动运行脚本时，在有文章的情况下能成功推送并发微信通知。
2. 手动运行脚本时，在无文章的情况下能发“无文章”通知。
3. 服务异常或接口报错时能发失败通知。
4. `crontab` 配置为：
   - `0 10 * * *`
   - `30 16 * * *`
5. 手工 `curl` 推送仍保持可用。

**实施结果**

交付物应包括：

- 一个自动推送 shell 脚本
- 一段新手可直接复制的 `crontab` 配置
- 一段服务器 `.env` 配置说明
- 一套本地/服务器手动验证命令
