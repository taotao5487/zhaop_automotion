# WeChat Download API 新手操作与维护指南

这份文档给第一次接手 `wechat-download-api` 的同事用，重点解决 4 件事：

1. 先把服务启动起来
2. 扫码登录并能订阅公众号
3. 手动触发 RSS 拉最新招聘文章
4. 导出筛选后的招聘，或者继续推到公众号草稿链路

如果你还需要维护医院官网招聘采集，请再看 [chongqing_medical_jobs_crawler/docs/新手维护指南.md](/Users/xiongtao/Documents/scaper_zhaopin/chongqing_medical_jobs_crawler/docs/新手维护指南.md)。

## 1. 这个项目是做什么的

`wechat-download-api` 负责：

- 搜索公众号，获取 `fakeid`
- 订阅公众号 RSS
- 定时或手动轮询最新文章
- 对文章做招聘筛选
- 导出 `confirmed` 招聘文章
- 在需要时接入官方预览包或共享草稿串

和 `chongqing_medical_jobs_crawler` 的分工是：

- 爬虫项目负责医院官网招聘采集和详情增强
- 本项目负责微信公众号文章拉取、招聘过滤、RSS 输出、招聘导出和草稿串推送

## 2. 启动前准备

项目根目录：`/Users/xiongtao/Documents/scaper_zhaopin/Wechat_api/wechat-download-api`

你先要认识这几个文件和配置：

- `.env`
  运行配置文件，登录成功后微信凭证也会自动回写到这里
- `env.example`
  `.env` 模板
- `data/rss.db`
  RSS 订阅、文章缓存、招聘导出、共享草稿串状态所在数据库
- `app.py`
  FastAPI 主入口

最关键的环境变量有 4 个：

### `SITE_URL`

作用：

- RSS 图片代理要用它来生成外部可访问地址

示例：

```env
SITE_URL=http://localhost:5001
```

如果这个值不对，RSS 阅读器里图片容易显示异常。

### `PROXY_URLS`

作用：

- 给微信公众号文章正文抓取提供 SOCKS5 代理池

建议：

- 开启完整正文抓取时尽量配置
- 少量订阅测试时可以先留空，但风控风险更高

### `RSS_FETCH_FULL_CONTENT`

作用：

- 是否对招聘候选文章继续抓完整正文

默认：

```env
RSS_FETCH_FULL_CONTENT=true
```

含义：

- `true`：会继续抓正文，筛选结果更准，但更依赖代理和登录状态
- `false`：只保留标题和摘要，风险更低，但筛选会更粗

### `.env` 初始化

第一次使用时先复制：

```bash
cp env.example .env
```

## 4. 怎么运行 RSS 程序得到最新招聘文章

这部分是新手最常用的主线。

### 4.0 从 RSS 拉文章到推送公众号草稿箱的完整流程

很多新手会误以为：

- 运行 RSS 程序后，文章会自动推到公众号草稿箱

实际上不是。

正确顺序是：

1. 启动服务
2. 扫码登录
3. 搜公众号并添加 RSS 订阅
4. 手动触发轮询，先把文章拉下来
5. 用导出接口确认哪些文章已经被判定为 `confirmed`
6. 再调用草稿接口，把这些已确认招聘推到公众号草稿箱

也就是说：

- `POST /api/rss/poll` 只是“拉文章”
- `GET /api/recruitment/export` 只是“看筛选结果”
- `POST /api/recruitment/official-draft/series/push` 才是“推到公众号草稿箱”

如果你只想照着一步一步做，最推荐下面这套：

```bash
# 1. 启动服务（当前推荐直接用 Docker 这套 5001 端口）
cd /Users/xiongtao/Documents/scaper_zhaopin/Wechat_api/wechat-download-api
docker-compose up -d

# 2. 确认服务健康
curl "http://localhost:5001/api/health"

# 3. 扫码登录后，搜索公众号拿 fakeid
curl "http://localhost:5001/api/public/searchbiz?query=人民日报"

# 4. 添加 RSS 订阅
curl -X POST http://localhost:5001/api/rss/subscribe \
  -H "Content-Type: application/json" \
  -d '{"fakeid":"MzA1MjM1ODk2MA==","nickname":"人民日报"}'

# 5. 手动拉一次最新文章
curl -X POST http://localhost:5001/api/rss/poll

# 6. 看看当前有哪些已确认、且还没推送过的招聘
curl "http://localhost:5001/api/recruitment/export?format=json&status=confirmed&push_status=unpushed&profile=title_url"

# 7. 推送最新一篇未推送的 confirmed 招聘到共享草稿串
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push"

# 8. 查看草稿串状态
curl "http://localhost:5001/api/recruitment/official-draft/series/status"
```

上面这套命令默认就是按当前 `docker-compose.yml` 的外部端口 `5001` 写的，最适合你直接在 VSCode 终端里复制测试。

### 4.1 在 VSCode 里直接测试的最短路径

如果你只是想在 VSCode 里把“拉消息 -> 过滤招聘 -> 推草稿箱”整条链路跑一遍，直接按下面顺序执行就行：

```bash
cd /Users/xiongtao/Documents/scaper_zhaopin/Wechat_api/wechat-download-api

# A. 确认服务活着
curl "http://localhost:5001/api/health"

# B. 确认已经登录
curl "http://localhost:5001/api/admin/status"

# C. 手动拉一次 RSS
curl -X POST "http://localhost:5001/api/rss/poll"

# D. 导出已筛成 confirmed、且还没推送过的招聘
curl "http://localhost:5001/api/recruitment/export?format=json&status=confirmed&push_status=unpushed&profile=title_url"

# E. 如果上一步确实有数据，再推到共享草稿串
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push"

# F. 查看共享草稿串状态
curl "http://localhost:5001/api/recruitment/official-draft/series/status"
```

最常见的判断方式是：

- 第 C 步成功，只代表“RSS 拉取动作执行了”
- 第 D 步有数据，才代表“已经筛出 confirmed 招聘”
- 第 E 步成功，才代表“招聘文章真正推进了公众号草稿箱链路”

如果第 D 步返回 `count=0`，通常有 3 种情况：

- 这次没有新招聘
- 有新文章，但还没被判成 `confirmed`
- 文章之前已经推送过，所以 `push_status=unpushed` 查不到

### 第 1 步：确保服务已经启动并登录

先确认服务能访问，再确认扫码登录已完成。

如果登录失效，后面的 RSS 拉取通常拿不到有效正文。

### 第 2 步：搜索公众号，拿到 `fakeid`

接口示例：

```bash
curl "http://localhost:5001/api/public/searchbiz?query=人民日报"
```

你要从返回结果里拿到目标公众号的 `fakeid`。

如果你现在按这份指南在 VSCode 里测试，默认就直接用 `5001`。

### 第 3 步：添加订阅

```bash
curl -X POST http://localhost:5001/api/rss/subscribe \
  -H "Content-Type: application/json" \
  -d '{"fakeid": "MzA1MjM1ODk2MA==", "nickname": "人民日报"}'
```

常见字段：

- `fakeid`：必填
- `nickname`：建议填，后面更好辨认
- `alias`、`head_img`：可选

### 第 4 步：手动触发一次轮询

```bash
curl -X POST http://localhost:5001/api/rss/poll
```

这里有两个关键认知：

- 轮询器服务启动后会自动运行
- 但 `POST /api/rss/poll` 只会处理“当前已到期的订阅”

所以如果你刚订阅、但这次还没到期，有可能返回“当前没有到期订阅”，这不是程序坏了。

### 第 5 步：读取 RSS 输出

读取某个公众号的 RSS：

```bash
curl "http://localhost:5001/api/rss/MzA1MjM1ODk2MA=="
```

读取全部订阅的聚合 RSS：

```bash
curl "http://localhost:5001/api/rss/all"
```

你也可以把这些地址直接加到 RSS 阅读器里。

## 5. 怎么导出筛选后的最新招聘

这个项目不是把所有公众号文章都当成招聘，而是会先筛选，再导出。

最常用的接口是：

```bash
curl "http://localhost:5001/api/recruitment/export?format=json&status=confirmed&push_status=unpushed&profile=title_url"
```

推荐参数含义：

- `status=confirmed`
  只导出已经确认是招聘的文章
- `push_status=unpushed`
  只看还没进草稿链路的文章
- `profile=title_url`
  只导出标题和原文链接，最适合新手快速筛查

如果你想下载 CSV：

```bash
curl -OJ "http://localhost:5001/api/recruitment/export?format=csv&status=confirmed&push_status=unpushed&profile=title_url"
```

### 5.1 导出之后，下一步怎么推送到公众号草稿箱

这一段单独强调一下，因为这是最容易断掉的地方。

你先做：

```bash
curl "http://localhost:5001/api/recruitment/export?format=json&status=confirmed&push_status=unpushed&profile=title_url"
```

这个返回结果的作用是：

- 帮你确认现在数据库里确实已经有 `confirmed` 招聘
- 帮你确认哪些文章还没有推送过
- 如果你想指定某一篇推送，可以从结果里复制它的 `source_url`

确认无误后，再进入推送步骤。

#### 方式 A：直接推送最新一篇未推送的 confirmed 招聘

```bash
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push"
```

这条命令适合：

- 你不在意具体是哪一篇
- 想让系统自动选择“最新一篇未进入共享草稿串的 confirmed 招聘”

#### 方式 B：推送指定文章

```bash
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push?source_url=https://example.com/recruit/123"
```

这条命令适合：

- 你已经从导出结果里人工确认过某一篇
- 想精确控制推送哪条文章

#### 方式 C：查看推送后状态

```bash
curl "http://localhost:5001/api/recruitment/official-draft/series/status"
```

你可以用它确认：

- 当前共享草稿串是否有 `media_id`
- 当前草稿里已经积累了多少篇
- 当前批次索引是多少

#### 推送前必须注意

草稿接口要真正写入微信官方草稿箱，需要 `.env` 里已经配置：

```env
OFFICIAL_WX_APPID=你的公众号appid
OFFICIAL_WX_APPSECRET=你的公众号appsecret
```

如果这两个值没配，推送草稿箱会失败。

## 6. 系统怎么判断“有用招聘”

这部分逻辑主要在：

- `utils/recruitment_filter.py`

新手先记住 4 条规则就够了。

### 6.1 标题优先命中，直接 `confirmed`

如果标题里直接命中明显招聘关键词，比如：

- `招聘`
- `公开招聘`
- `招聘公告`
- `人才招聘`
- `人才引进`

系统会优先判成招聘，状态通常直接进入 `confirmed`。

### 6.2 标题粗筛命中后，会继续抓正文再二次确认

如果标题只是粗略像招聘，但还不够稳，系统会：

1. 先把它记成招聘候选
2. 在 `RSS_FETCH_FULL_CONTENT=true` 时继续抓完整正文
3. 再结合正文里的 `岗位`、`报名`、`应聘`、`面试` 等信息确认

所以你会看到“标题先过一轮，正文再过一轮”的效果。

### 6.3 命中排除词会 `rejected`

如果文章更像这些内容：

- 会议
- 讲座
- 培训
- 科普
- 活动预告
- 新闻速递

即使抓到了，也会更倾向于被排掉。

### 6.4 边界情况会进 `manual_review`

有一种特殊情况是：

- 标题里有招聘线索
- 文章又带图片
- 但正文文本很短，短到不足以自动确认

这类文章可能会进入 `manual_review`，意思是它看起来像招聘，但程序不想武断地下结论，留给人工再看。

## 7. 草稿与预览

如果你后续不只想看招聘结果，还要进入公众号内容链路，可以关注这两块。

### 7.1 官方预览包

手动生成官方预览包：

```bash
curl -X POST "http://localhost:5001/api/recruitment/official-preview/run?limit=100"
```

查看预览状态：

```bash
curl "http://localhost:5001/api/recruitment/official-preview/status"
```

这条链路更适合“先本地生成预览产物，再人工看效果”。

### 7.2 共享草稿串

把最新一篇未进入共享草稿串的 confirmed 招聘推入草稿链路：

```bash
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push"
```

推送指定原文：

```bash
curl -X POST "http://localhost:5001/api/recruitment/official-draft/series/push?source_url=https://example.com/recruit/123"
```

查看共享草稿串状态：

```bash
curl "http://localhost:5001/api/recruitment/official-draft/series/status"
```

适用场景：

- 你已经确认筛选结果没问题
- 想把 confirmed 招聘逐步积累进同一个共享草稿批次

### 7.3 RSS 推公众号时，最实用的理解方式

新手可以把这条链路理解成：

1. `searchbiz` 找公众号
2. `/api/rss/subscribe` 加订阅
3. `/api/rss/poll` 拉文章
4. `/api/recruitment/export?...status=confirmed...` 看筛选结果
5. `/api/recruitment/official-draft/series/push` 推进共享草稿串

前 4 步是“把招聘找出来”。

第 5 步才是“把招聘写进公众号草稿箱”。

## 8. 日常巡检怎么做

如果你是日常维护，不一定每天改代码，但建议每天做这几类检查。

### 8.1 检查登录是否过期

最简单的办法：

- 打开管理页或登录页确认状态
- 看最近 RSS 拉取是否还正常

如果登录过期，就重新扫码。

### 8.2 检查轮询器状态

```bash
curl "http://localhost:5001/api/rss/status"
```

重点看：

- `running`
- `subscription_count`
- 当前轮询器是否真的在跑

### 8.3 检查订阅列表

```bash
curl "http://localhost:5001/api/rss/subscriptions"
```

这能帮你确认：

- 订阅有没有加进去
- RSS 地址是否正确
- 当前缓存文章数是否有增长

### 8.4 检查导出结果

推荐定期看：

```bash
curl "http://localhost:5001/api/recruitment/export?format=json&status=confirmed&push_status=unpushed&profile=title_url"
```

这一步适合确认：

- 最近确实有新的 confirmed 招聘
- 是否还存在很多未推送文章

## 9. 常见问题

### 登录失效

现象：

- 手动拉取拿不到有效结果
- 正文抓取失败
- 轮询日志里提示未登录或需要验证

处理：

- 重新打开 `login.html`
- 用公众号管理员微信重新扫码

### `POST /api/rss/poll` 提示没有新文章

先区分两种情况：

1. 当前没有到期订阅
2. 到期了，但这轮确实没有新文章

第一种最常见，不一定是故障。

### 没有配置代理，导致风控或正文抓取不稳定

现象：

- 文章正文抓取容易失败
- 登录环境更容易触发验证

处理：

- 在 `.env` 里补 `PROXY_URLS`
- 如果只是临时测试，也可以先把 `RSS_FETCH_FULL_CONTENT=false`

### `SITE_URL` 配错导致图片显示异常

现象：

- RSS 阅读器里文字正常，但图片打不开

处理：

- 把 `.env` 里的 `SITE_URL` 改成真实访问地址
- 当前这台机器按 Docker 暴露端口就是 `http://localhost:5001`
- 局域网或 Docker 暴露端口时，要改成实际对外访问地址和端口

## 10. 新手推荐操作顺序

如果你今天第一次维护这个项目，建议按下面顺序来：

1. `cp env.example .env`
2. 启动服务
3. 打开 `login.html` 扫码登录
4. 用搜索接口拿一个公众号的 `fakeid`
5. 调用 `POST /api/rss/subscribe`
6. 调用 `GET /api/rss/status`
7. 调用 `POST /api/rss/poll`
8. 读取 `/api/rss/{fakeid}` 或 `/api/rss/all`
9. 用 `/api/recruitment/export?...status=confirmed...` 看筛选结果
10. 调用 `POST /api/recruitment/official-draft/series/push`
11. 调用 `GET /api/recruitment/official-draft/series/status` 确认是否推送成功

这样最容易把完整链路跑通。
